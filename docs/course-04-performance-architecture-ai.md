## 第一部分：性能优化

### 一、性能优化的价值

#### 用户体验提升

性能直接影响用户体验——启动耗时过长、滑动卡顿、视频起播速度慢等，这些都是显而易见的。

#### 业务价值

性能优化需要投入专业研发人力，成本高昂。需要通过数据衡量性能优化对业务的具体价值：

- **相关性分析**：分析核心性能指标提升 1% 后对业务指标的影响，结论是核心性能指标与核心业务指标之间具有较高相关性：

| 性能指标 | 人均播放时长 | 人均使用时长 | DAU |
|---------|:----------:|:----------:|:---:|
| 启动耗时 | xx% | xx% | xx% |
| 滑动流畅性 | xx% | xx% | xx% |
| ANR 率 | xx% | xx% | xx% |

#### 字节在性能优化方面的投入

| | 公司 APP 基建团队 | 业务研发团队 |
|---|---|---|
| **投入** | 性能监控工具（流畅性、ANR）、通用优化能力（CPU 提频、GC 抑制） | 大型 APP 有成建制团队全职做性能优化 |
| **产出** | 核心性能指标行业领先 | 每年对核心业务指标有 x% 提升 |

### 二、如何衡量性能的好坏

#### 常见核心性能指标

| 类别 | 指标 |
|------|------|
| 启动 | 启动耗时 |
| UI 流畅性 | 帧率（FPS）、丢帧率 |
| 稳定性 | ANR 率 |
| 视频业务 | 可感知首帧耗时、起播失败率、卡顿率 |
| 页面 | 页面打开耗时 |
| 图片 | 图片加载耗时 |

#### 流畅性基础认知

- 屏幕每秒刷新 60 次，每帧间隔 **16.6ms**
- APP 需要保证每帧在 16.6ms 内渲染完成，否则出现卡顿（Jank）
- Android 使用**三缓冲区模型**：GPU 在后缓冲区完成渲染后等待 VSync，避免空闲浪费
- 实际上业务复杂时做到 0 丢帧几乎不可能，平均 **57 帧以上就很流畅**

#### 性能数据采集方法

**APP 启动 / 页面打开耗时**：

```
LaunchCost = RenderSucceedTime - ClickTime
```

- 起点：用户点击时刻（二级页面）或 APP 启动代码最早点（冷启动）
- 终点：页面渲染成功。通用方式是在触发渲染后 post 一个消息（消息屏障机制保证下一帧渲染完成）；含图页面需等所有图片渲染完成；视频页面需等播放器首帧渲染成功

**UI 流畅性**：

| 指标 | 定义 |
|------|------|
| **帧率（FPS）** | FPS = 总帧数 / 统计时间。高 FPS 不绝对等于流畅（可能前 800ms 流畅但最后 200ms 严重卡顿） |
| **丢帧率** | 丢 N 帧率 = 丢 N 帧次数 / 统计时长。分为不同 level（如丢 3 帧率、丢 5 帧率） |

采集原理：Android View 渲染体系存在 `FrameCallback` 接口，可监听每帧绘制时机。

**ANR**：

| 项目 | 说明 |
|------|------|
| 定义 | Application Not Responding，应用无响应。系统对特定事件响应进行超时检测 |
| 触发场景 | Input 事件超时、Service 启动超时、广播处理超时、ContentProvider 启动超时 |
| 主要原因 | APP UI 线程卡死（占绝大多数） |

采集方式对比：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **watch_dog** | 子线程定时向主线程发心跳消息，超时视为 ANR | 简单可靠、无入侵性、兼容性好 | 准确率不高 |
| **捕获 SIGQUIT** | ANR 时 AMS 给进程发 SIGQUIT 信号 | 准确性高 | 方案复杂、需 HOOK 系统方法、兼容性风险 |

#### 收益分析手段 — AB 实验

> Talk is cheap, show me the data.

在线上用户中随机选取一批用户，随机分给原策略 A 和新策略 B，结合统计方法得出两种策略相对效果的准确估计。样本量越多，实验越灵敏和准确。

### 三、常见性能优化方法

#### 基础优化思想

| 策略 | 说明 | 示例 |
|------|------|------|
| **正面优化** | 从逻辑、算法层面优化代码执行时长 | 更优解压算法、更高压缩率图片 |
| **预加载** | 空间换时间，设备内存充足时提前加载 | 缓存首屏数据、预加载下个视频数据 |
| **按需/延迟加载** | 不立即使用的模块延后加载，选合适折中时机 | 西瓜直播组件：冷启动不加载，直播活跃用户启动后加载 |
| **异步化** | 非 UI 逻辑放子线程 | 文件读写、数据库、网络请求、图像处理、编解码、加解密 |
| **复用** | 避免频繁 GC 导致卡顿，大对象复用 | View 复用、播放器复用、Fragment 复用 |

#### 常用工具

| 工具 | 作用 | 来源 |
|------|------|------|
| **perfetto** | 生成火焰图，分析代码执行耗时 | Google 官方 |
| **btrace** | 代码火焰图分析 | 字节开源 |
| **Layout Inspector** | 视图层级与属性分析 | Android Studio 自带 |
| **CodeLocator** | 视图分析工具 | 字节开源 |
| **MAT / Memory Profiler** | 内存分析与泄露排查 | Eclipse / Android Studio |

#### 性能优化手段 — 基础篇

**各种预加载**：

| 预加载对象 | 方式 |
|-----------|------|
| **View** | 提前加载 View 对象、提前渲染（抖音 Feed 提前渲染好屏幕外下一个卡片） |
| **视频** | 提前下载视频数据 + 提前创建播放器进行首帧渲染 → 极速起播 |
| **图片** | 提前下载 + 解码到 Bitmap 备用 |
| **代码** | 提前触发类加载，提升首次使用性能 |

预加载时机：不当预加载可能造成当前体验负向（如挤占网络资源导致当前视频卡顿）。至少满足：列表不在滑动 + 当前视频已起播且有缓冲进度。

**布局优化**：

| 问题 | 优化方式 |
|------|---------|
| **过度绘制** | 优化层级减少嵌套、去掉不必要背景色、不可见 View 设为 GONE |
| **XML inflate 耗时** | 子线程 inflate，最好提前创建 |
| **重复 measure** | 理解布局测量算法、减少嵌套、使用 ConstraintLayout |

**内存优化**：

- 内存泄露治理：静态变量 + 监听器管理 + 监控系统
- 对象复用

#### 性能优化手段 — 进阶篇

**系统级优化**：

| 手段 | 原理 |
|------|------|
| **CPU 提频** | OS 为省电动态调节 CPU 频率，在关键阶段（启动、起播、滑动）提频 |
| **GC 抑制** | 启动、滑动、二级页面启动等关键阶段抑制 GC（控制范围内） |
| **线程优先级** | 主线程绑定大核、视频解码线程提升优先级 |

**精细化**：

进入深水区后，需要找到精细化维度，在不同场景生效不同策略：

| 维度 | 策略示例 |
|------|---------|
| **人群** | 直播活跃用户提前加载直播组件、大龄用户简化交互 UI |
| **设备性能** | 低端机禁止动画、禁止高斯模糊 |
| **网络环境** | 网速差时加载低清晰度视频和图片 |

## 第二部分：架构设计

### 一、架构的概念、目标和价值

#### 什么是架构

架构是指代码的组织形式，包含**模块和层次的划分**，模块之间的**交互接口设计**等。

#### 架构设计的目标

| 目标 | 说明 |
|------|------|
| **可维护性** | 清晰架构更易阅读和理解，问题定位和修复更快 |
| **可迭代性** | 新功能开发免于对整体架构的修改 |
| **分工协作** | 不同模块由不同人负责，并行开发，"术业有专攻" |
| **复用性** | 模块/层次维度复用，提升开发效率 |

#### 架构设计的价值

- **基础层次**：长期的效率与质量
- **高层次**：为系统演进提供更大可能——每个功能模块做到更深层次水平

---

### 二、架构基础知识

#### SOLID 原则

| 原则 | 定义 | 理解 |
|------|------|------|
| **SRP** 单一职责 | 任何软件模块只对某一类行为负责 | 最基础要求：拆分为不同模块，各负责一个功能 |
| **OCP** 开闭原则 | 软件应易于扩展，抗拒修改 | 抽象稳定主体框架，新功能以插件方式扩展 |
| **LSP** 里氏替换 | 子类型必须能替换父类型 | 接口的实现要严格按接口定义来 |
| **ISP** 接口隔离 | 客户端不应被迫依赖不使用的接口 | 接口拆分为更小更具体的接口 |
| **DIP** 依赖倒置 | 多依赖抽象，不依赖具体实现 | 业务逻辑依赖接口 IDB，不直接依赖具体数据库 |

**依赖倒置详解**：

```
正常依赖：  业务逻辑 → 数据库（耦合，难替换）
依赖倒置：  业务逻辑 → 接口 IDB ← 数据库实现（解耦，可替换）
```

#### 设计模式（23 种）

| 分类 | 模式 |
|------|------|
| **创建型** (5) | 单例、工厂方法、抽象工厂、建造者、原型 |
| **结构型** (7) | 适配器、桥接、组合、装饰器、外观、享元、代理 |
| **行为型** (11) | 职责链、命令、解释器、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者 |

#### 分层理念

- **定义**：根据依赖关系，从垂直方向切分软件系统为不同层次。上层依赖下层，不能反向依赖，最好不跨层依赖。
- **浅层次作用**：分工与复用
- **深层次作用**：让巨人的肩膀更坚实——底层高可靠性、高复杂度、低变化频率

> 在字节，基于公司内丰富的基础能力库，一个新的 APP 可以由一个很小的团队在很快的时间内开发出来。

---

### 三、常见架构范式

#### MVC、MVP、MVVM

| 层 | 职责 |
|----|------|
| **Model** | 数据管理和业务逻辑，与 UI 层完全解耦 |
| **View** | UI 展示和用户交互 |
| **中间层** | 联动 Model 和 View（三种架构的主要区别） |

核心好处：
- 业务逻辑可复用，适应不同视图变化
- 便于对业务逻辑进行自动化测试

#### 组件化理念

对高复杂度、高复用价值的软件模块，做到**高内聚、低耦合**，提升可复用性。

组件对外接口一般分为四部分：
1. **常规功能性接口**
2. **事件回调**：外部关心组件内部事件时
3. **配置**：开关可选功能、修改运行参数
4. **外部依赖定义**：不适合内聚的能力让宿主通过依赖注入实现

**播放器组件化案例**：

非组件化设计的问题：
- 耦合特定业务场景 UI 逻辑，无法直接复用
- 多场景差异处理导致复杂度爆炸
- 存在冗余运行开销，性能差
- 多方人员共同开发同一份代码，协作困难

组件化设计：
- 播放行为差异通过 config 配置
- 事件回调供业务场景实现功能
- 业务场景持有播放器组件，自行开发 UI 和业务逻辑

#### 插件化架构

大多数成熟的复杂系统，最终都会发展出插件化架构。

**为什么需要插件化**：
- 避免核心系统代码变得复杂难以维护
- 避免核心系统被破坏
- 避免代码冲突，利于协作

**插件化架构运行方式**：
- 设计标准化交互接口，让核心系统与插件交互
- **插件被调用** → 抽象为**事件**
- **插件调用核心系统** → 抽象为 **Context**

### 四、架构案例：西瓜 Feed 容器架构

#### 整体三层

| 层 | 内容 |
|----|------|
| **Feed 业务层** | 插件方式注入，与核心系统无关的业务逻辑 |
| **Feed 容器层** | 核心的数据加载、视图渲染、UI 交互 + 插件式注入能力 |
| **基础能力层** | 基础能力组件 |

#### 容器层架构

垂直切分：
- **数据层**
- **逻辑层**（拆分为多个不同职责的类）
- **视图层**

容器层复用性：抽象依赖为接口，支持不同场景配置不同实现（数据源、列表视图、数据刷新策略）

#### 业务场景应用

多个不同业务场景的 Feed 都复用同一套容器代码。差异通过配置或插件化框架解决：

| 方式 | 解决的差异点 |
|------|------------|
| **配置** | 数据源不同、列表 UI 差异、数据刷新逻辑差异 |
| **插件化框架** | 卡片类型不同、页面业务逻辑不同 |

## 第三部分：AI 在客户端的应用概述

### 一、传统模型

#### 图像、语音类算法

传统基于硬规则的算法已被深度学习取代：
- 语音识别
- 视频创作中的图像/声音处理
- 人脸识别
- 美颜
- **超分（超分辨率）**：把低分辨率图像/视频转变为高分辨率

#### 复杂决策逻辑

很多复杂策略无法通过硬规则实现，依赖端智能：

| 应用场景 | 说明 |
|---------|------|
| 用户意图识别 | 评论预加载：判断用户是否会打开评论 |
| 视频卡顿预测 | 提前预判卡顿 |
| 用户左右手识别 | 适配不同交互习惯 |

**技术方案 — 端云协同**：
1. 客户端采集特征数据 → 上传云端 → 训练模型 → 形成算法包
2. 端上部署推理模型 → 查询算法包 → 触发推理

---

### 二、LLM（大语言模型）

大语言模型在端上的部署和应用目前还在初期探索中，未来有机会在端内直接实现：

- 内容创作 AI 辅助
- 文字、图片、视频内容总结
- 更强大的语音助手，实现更复杂更准确的用户意图识别
- 实时语音翻译
- 播放器实时字幕、实时翻译
